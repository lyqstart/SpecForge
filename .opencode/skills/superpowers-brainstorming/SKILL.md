# Superpowers Brainstorming

## 指令

在开始撰写需求之前，你**必须**从以下 8 个维度逐一进行头脑风暴，每个维度至少列出一个考虑点：

### 1. 业务需求
- 核心业务价值是什么？
- 目标用户场景有哪些？
- 涉及哪些业务流程？
- 与现有业务的关系如何？

### 2. 技术约束
- 平台有哪些限制？
- 依赖关系是什么？
- 兼容性要求有哪些？
- 技术栈的选择约束？

### 3. 用户体验
- 交互方式是否直观？
- 反馈机制是否及时？
- 错误提示是否友好？
- 学习成本是否可接受？

### 4. 安全合规
- 权限控制如何设计？
- 数据保护措施有哪些？
- 审计要求是什么？
- 是否涉及敏感信息处理？

### 5. 运维部署
- 部署方式是什么？
- 监控需求有哪些？
- 故障恢复策略是什么？
- 日志和可观测性如何保障？

### 6. 成本预算
- 资源消耗预估如何？
- API 调用成本是否可控？
- 存储成本如何？
- 是否有性能与成本的权衡？

### 7. 扩展性
- 未来扩展方向是什么？
- 接口是否预留了扩展点？
- 版本兼容性如何保障？
- 是否支持渐进式增强？

### 8. EARS 模式覆盖
在开始编写需求之前，逐一考虑以下六种 EARS（Easy Approach to Requirements Syntax）模式，确保需求覆盖各种行为类型：

- **Ubiquitous（始终生效）**：是否有行为应始终处于活动状态，无需任何触发条件？
  - 例：系统始终应保持的不变量、全局约束
- **Event-driven（事件触发）**：哪些事件会触发该需求描述的行为？
  - 例：用户操作、外部信号、定时触发
- **State-driven（状态依赖）**：是否有系统状态会影响行为的生效与否？
  - 例：系统处于维护模式时、用户已登录时
- **Optional-feature（可选功能）**：该功能是否可选或可配置？是否依赖某个开关或许可？
  - 例：付费功能、实验性功能、管理员启用的选项
- **Unwanted-behavior（异常处理）**：可能出错的情况有哪些？系统应如何恢复？
  - 例：网络超时、输入非法、资源不足
- **Complex（组合条件）**：是否有需求需要组合多个条件（状态 + 事件、可选 + 状态等）？
  - 例：当功能启用且系统处于特定状态时，收到某事件后执行操作

**引导问题清单：**
1. 有哪些行为应始终生效，无需任何前置条件？→ Ubiquitous
2. 哪些事件会触发系统响应？→ Event-driven
3. 系统的哪些状态会影响行为？→ State-driven
4. 该功能是否可选或可配置？→ Optional-feature
5. 可能出错的情况有哪些？系统应如何恢复？→ Unwanted-behavior
6. 是否有需求需要组合两个或以上条件？→ Complex

## 规则

1. **必须完成所有 8 个维度的头脑风暴后，才能开始撰写结构化需求。**
2. 每个维度至少列出一个具体的考虑点，不得留空。
3. 如果某个维度确实不适用，必须明确说明原因（例如："成本预算：本功能不涉及外部 API 调用，无额外成本"）。
4. 头脑风暴的结果应作为需求分析的输入，确保最终需求文档覆盖了各维度的关键考虑。
