## Error Code Reference

### Error Code Structure

All Scope Gate errors follow this structure:

```typescript
interface ScopeError {
  code: string;           // Error code (e.g., "SCOPE_BOUNDARY_VIOLATION")
  message: string;        // Human-readable error message
  capabilityId: string;   // ID of the capability that caused the error
  scopeTag: "p0" | "p1" | "p2";  // Scope tag of the capability
  requiredFlag?: string;  // Feature flag needed to enable the capability
}
```

### Error Code Catalog

| Error Code | Error Class | When Thrown | HTTP Status Equivalent |
|------------|-------------|-------------|------------------------|
| **SCOPE_BOUNDARY_VIOLATION** | `ScopeBoundaryViolationError` | P1/P2 capability used in V6.0 without feature flag | 403 Forbidden |
| **CAPABILITY_UNAVAILABLE** | `CapabilityUnavailableError` | Capability not registered, dependencies missing, or general unavailability | 404 Not Found |
| **DEPENDENCY_VIOLATION** | `DependencyError` | P0 capability depends on P1/P2 capability | 409 Conflict |
| **CONFIGURATION_ERROR** | `ConfigurationError` | Invalid or missing configuration | 400 Bad Request |

### Detailed Error Descriptions

#### SCOPE_BOUNDARY_VIOLATION

**Error Class**: `ScopeBoundaryViolationError`

**When Thrown**:
- Attempting to use a P1 capability in V6.0 release branch without enabling its feature flag
- Attempting to use a P2 capability in V6.0 release branch without enabling its feature flag
- Attempting to use a P1/P2 capability in production environment without explicit enablement

**Error Message Examples**:
```
P1 capability 'knowledge-graph' is not available in V6.0 release branch. 
Enable feature flag 'enable_knowledge-graph' to use this capability.
```

```
P2 capability 'multimodal' is not available in V6.0 release branch. 
Enable feature flag 'enable_multimodal' to use this capability.
```

**Resolution**:
1. Check if you're in the correct environment (development/staging may allow P1/P2)
2. Enable the required feature flag: `feature-flag enable enable_<capability-id>`
3. If in production, verify the capability is approved for use
4. Consider using a P0 alternative if available

#### CAPABILITY_UNAVAILABLE

**Error Class**: `CapabilityUnavailableError`

**When Thrown**:
- Capability is not registered in the Scope Registry
- Required dependencies are not available
- General unavailability due to system state
- Feature flag is disabled for a P1/P2 capability

**Error Message Examples**:
```
Capability 'workflow-runtime' is currently unavailable. 
Enable feature flag 'enable_workflow-runtime' to use this capability.
```

```
Capability 'unknown-capability' is currently unavailable. 
Contact your administrator for access.
```

**Resolution**:
1. Verify the capability ID is correct
2. Check if the capability is registered: `capability-list --scope all`
3. Enable required feature flags: `feature-flag list`
4. Check dependency availability
5. Contact system administrator if issue persists

#### DEPENDENCY_VIOLATION

**Error Class**: `DependencyError`

**When Thrown**:
- P0 capability depends on a P1 capability
- P0 capability depends on a P2 capability
- P1 capability depends on a P2 capability (in V6.0)

**Error Message Examples**:
```
Capability 'my-p0-feature' cannot be used because it depends on 'knowledge-graph' which is P1
```

**Resolution**:
1. Redesign the capability to remove P1/P2 dependencies
2. Move the capability to a later scope (P1/P2)
3. Enable the dependent capability via feature flag (if appropriate)
4. Consider implementing a P0 alternative

#### CONFIGURATION_ERROR

**Error Class**: `ConfigurationError`

**When Thrown**:
- Missing or invalid configuration file
- Invalid scope tag in `.config.kiro` file
- Missing required configuration keys
- Configuration schema validation failure

**Error Message Examples**:
```
Invalid configuration: Missing 'scopeTag' field in .config.kiro
```

```
Configuration error: Invalid scope tag 'p3' in .config.kiro
```

**Resolution**:
1. Verify configuration file exists and is valid JSON
2. Check `.config.kiro` files for correct `scopeTag` values
3. Run validation: `scope-validate --path ./packages`
4. Review configuration schema in design documentation

## Troubleshooting Guide

### Quick Diagnosis Flowchart

```
Scope Error Encountered
├── Error code = SCOPE_BOUNDARY_VIOLATION?
│   ├── Yes → Check current release branch (`scope-context`)
│   │   ├── V6.0? → Enable feature flag (`feature-flag enable`)
│   │   └── Other? → Check environment defaults
│   └── No → Continue
├── Error code = CAPABILITY_UNAVAILABLE?
│   ├── Yes → Check capability registration (`capability-list`)
│   │   ├── Registered? → Check dependencies
│   │   └── Not registered? → Register capability
│   └── No → Continue
├── Error code = DEPENDENCY_VIOLATION?
│   ├── Yes → Check dependency graph
│   │   ├── P0 depends on P1/P2? → Redesign or enable flag
│   │   └── Other? → Review scope boundaries
│   └── No → Continue
└── Error code = CONFIGURATION_ERROR?
    ├── Yes → Validate configuration (`scope-validate`)
    └── No → Unknown error, check logs
```

### Step-by-Step Troubleshooting

#### Step 1: Identify the Error

```bash
# 1. Get error details
echo "Error Code: $ERROR_CODE"
echo "Capability: $CAPABILITY_ID"
echo "Message: $ERROR_MESSAGE"

# 2. Check current scope context
bun run packages/scope-gate/bin/scope-context.ts
```

#### Step 2: Check Capability Status

```bash
# 1. List all capabilities
bun run packages/scope-gate/bin/capability-list.ts

# 2. Filter for specific capability
bun run packages/scope-gate/bin/capability-list.ts | grep "$CAPABILITY_ID"

# 3. Check feature flags
bun run packages/scope-gate/bin/feature-flag.ts list | grep "$CAPABILITY_ID"
```

#### Step 3: Validate Configuration

```bash
# 1. Run scope validation
bun run packages/scope-gate/bin/scope-validate.ts

# 2. Check specific package
bun run packages/scope-gate/bin/scope-validate.ts --path "./packages/$MODULE_NAME"
```

#### Step 4: Review Logs

```bash
# 1. Check audit logs (if configured)
cat ./logs/scope-audit.log | tail -20

# 2. Check application logs
cat ./logs/app.log | grep -i "scope" | tail -20
```

### Common Error Patterns

#### Pattern 1: P1/P2 Capability Not Working in Development

**Symptoms**:
- `SCOPE_BOUNDARY_VIOLATION` error in development environment
- Feature flag appears enabled but capability still unavailable

**Diagnosis**:
```bash
# Check environment
echo $SCOPEGATE_ENVIRONMENT
echo $NODE_ENV

# Check release branch
echo $SCOPEGATE_RELEASE_BRANCH

# Verify feature flag
bun run packages/scope-gate/bin/feature-flag.ts list --verbose
```

**Solution**:
```bash
# Set environment to development
export SCOPEGATE_ENVIRONMENT=development
export NODE_ENV=development

# Enable the capability
bun run packages/scope-gate/bin/feature-flag.ts enable "enable_$CAPABILITY_ID" --reason "Development testing"

# Verify
bun run packages/scope-gate/bin/scope-context.ts
```

#### Pattern 2: Capability Missing from Registry

**Symptoms**:
- `CAPABILITY_UNAVAILABLE` error for known capability
- Capability not listed in `capability-list` output

**Diagnosis**:
```bash
# Check if REQ-25 is loaded
bun run packages/scope-gate/bin/scope-context.ts --output json | jq '.sourceInfo.capabilitiesLoaded'

# Check parent spec path
bun run packages/scope-gate/bin/scope-context.ts --output json | jq '.sourceInfo.parentSpecPath'
```

**Solution**:
```bash
# 1. Verify REQ-25 contains the capability
grep -i "$CAPABILITY_ID" .kiro/specs/v6-architecture-overview/requirements.md

# 2. Manually register capability (temporary fix)
import { ScopeRegistry } from '@specforge/scope-gate';
const registry = new ScopeRegistry();
registry.registerCapability({
  id: '$CAPABILITY_ID',
  scopeTag: 'p1', // or appropriate tag
  // ... other fields
});

# 3. Report missing capability to spec maintainers
```

#### Pattern 3: Configuration Validation Failure

**Symptoms**:
- `CONFIGURATION_ERROR` on application startup
- `.config.kiro` validation failures

**Diagnosis**:
```bash
# Validate all specs
bun run packages/scope-gate/bin/scope-validate.ts --output json | jq '.specScopeTags'

# Check specific .config.kiro file
cat .kiro/specs/$SPEC_NAME/.config.kiro | jq '.scopeTag'
```

**Solution**:
```bash
# 1. Fix invalid scope tag
# In .config.kiro:
{
  "scopeTag": "p0",  # Must be "p0", "p1", or "p2"
  # ... other fields
}

# 2. Run validation to confirm fix
bun run packages/scope-gate/bin/scope-validate.ts --path ".kiro/specs/$SPEC_NAME"
```

## Common Scenarios

### Scenario 1: Enabling P1 Capability for Testing

**Goal**: Enable a P1 capability in development environment

**Steps**:
```bash
# 1. Check current state
bun run packages/scope-gate/bin/scope-context.ts

# 2. Enable the capability
bun run packages/scope-gate/bin/feature-flag.ts enable "enable_knowledge-graph" \
  --reason "Q2 testing" \
  --user "developer-1"

# 3. Verify enablement
bun run packages/scope-gate/bin/capability-list.ts --scope p1

# 4. Test the capability
# (Your test code here)

# 5. Disable after testing
bun run packages/scope-gate/bin/feature-flag.ts disable "enable_knowledge-graph" \
  --reason "Testing complete" \
  --user "developer-1"
```

### Scenario 2: Debugging Scope Violation in CI/CD

**Goal**: Fix scope violation that only occurs in CI/CD pipeline

**Steps**:
```bash
# 1. Reproduce locally with CI environment
export SCOPEGATE_ENVIRONMENT=production
export SCOPEGATE_RELEASE_BRANCH=v6.0

# 2. Run validation
bun run packages/scope-gate/bin/scope-validate.ts

# 3. Check for P0 depending on P1/P2
bun run packages/scope-gate/bin/scope-validate.ts --output json | \
  jq '.codeDependencies[] | select(.type == "error")'

# 4. Fix the dependency issue
# Option A: Remove P1/P2 dependency
# Option B: Add feature flag guard
# Option C: Move capability to appropriate scope

# 5. Re-run validation
bun run packages/scope-gate/bin/scope-validate.ts
```

### Scenario 3: Migrating Capability from P1 to P0

**Goal**: Move a capability from P1 to P0 scope

**Steps**:
```bash
# 1. Update REQ-25 in parent spec
# Move capability from P1 list to P0 list

# 2. Update .config.kiro files
# Change scopeTag from "p1" to "p0"

# 3. Remove feature flag guards (if any)
# Remove @guardCapability decorators

# 4. Run comprehensive validation
bun run packages/scope-gate/bin/scope-validate.ts --path ./packages

# 5. Verify capability is now P0
bun run packages/scope-gate/bin/capability-list.ts --scope p0 | grep "$CAPABILITY_ID"
```

### Scenario 4: Handling Missing Parent Spec

**Goal**: Resolve "Parent spec not found" errors

**Steps**:
```bash
# 1. Check parent spec path
bun run packages/scope-gate/bin/scope-context.ts --output json | \
  jq '.sourceInfo.parentSpecPath'

# 2. Verify parent spec exists
ls -la .kiro/specs/v6-architecture-overview/requirements.md

# 3. Set custom parent spec path (if needed)
export SCOPEGATE_PARENT_SPEC_PATH=/custom/path/to/requirements.md

# 4. Or specify at runtime
const registry = new ScopeRegistry();
await registry.loadFromParentSpec('/custom/path/to/requirements.md');

# 5. Verify capabilities are loaded
bun run packages/scope-gate/bin/capability-list.ts
```

## Debugging Tools

### Built-in CLI Tools

| Tool | Purpose | Debugging Use |
|------|---------|---------------|
| `scope-context` | Inspect current scope | Verify environment, release branch, flags |
| `capability-list` | List capabilities | Check registration and status |
| `feature-flag` | Manage flags | Enable/disable capabilities for testing |
| `scope-validate` | Validate scope | Find violations and misconfigurations |

### Environment Variables for Debugging

```bash
# Enable debug logging
export DEBUG=scope-gate:*

# Override release branch
export SCOPEGATE_RELEASE_BRANCH=development

# Override environment
export SCOPEGATE_ENVIRONMENT=development

# Custom config path
export SCOPEGATE_CONFIG_PATH=./debug-config.json

# Enable audit logging
export SCOPEGATE_AUDIT_LOG=./debug-audit.log
```

### Programmatic Debugging

```typescript
import { 
  ScopeRegistry, 
  RuntimeScopeChecker,
  FeatureFlagManager 
} from '@specforge/scope-gate';

async function debugScopeIssue() {
  // 1. Create registry with debug logging
  const registry = new ScopeRegistry();
  
  // 2. Load capabilities with error handling
  try {
    await registry.loadFromParentSpec();
  } catch (error) {
    console.error('Failed to load capabilities:', error);
    // Check parent spec path, format, permissions
  }
  
  // 3. Create checker with debug context
  const checker = new RuntimeScopeChecker(registry, {
    releaseBranch: 'v6.0',
    featureFlags: new Set(),
    environment: 'production'
  });
  
  // 4. Test specific capability
  try {
    checker.checkCapability('problematic-capability', checker.getCurrentContext());
    console.log('Capability is available');
  } catch (error) {
    console.error('Scope error:', error);
    console.error('Error code:', error.code);
    console.error('Required flag:', error.requiredFlag);
  }
  
  // 5. Check all capabilities
  const results = checker.checkAll(checker.getCurrentContext());
  const unavailable = results.filter(r => !r.available);
  console.log('Unavailable capabilities:', unavailable);
}
```

## Best Practices

### Error Handling Best Practices

1. **Always Catch Scope Errors**:
   ```typescript
   try {
     checker.checkCapability(capabilityId, context);
   } catch (error) {
     if (error instanceof ScopeBoundaryViolationError) {
       // Handle scope violation
       logger.warn(`Scope violation: ${error.capabilityId}`);
       return { error: 'FEATURE_UNAVAILABLE', suggestion: error.message };
     }
     // Re-throw unexpected errors
     throw error;
   }
   ```

2. **Provide User-Friendly Messages**:
   ```typescript
   function formatScopeError(error: ScopeError): UserMessage {
     switch (error.code) {
       case 'SCOPE_BOUNDARY_VIOLATION':
         return {
           title: 'Feature Not Available',
           message: `The ${error.capabilityId} feature is not available in this version.`,
           action: error.requiredFlag 
             ? `Enable feature flag '${error.requiredFlag}' to use this feature.`
             : 'Contact your administrator for access.'
         };
       // ... other cases
     }
   }
   ```

3. **Log for Debugging**:
   ```typescript
   catch (error) {
     if (error instanceof ScopeError) {
       auditLogger.logViolationAttempt({
         capabilityId: error.capabilityId,
         scopeTag: error.scopeTag,
         context: currentContext,
         stackTrace: error.stack,
         timestamp: new Date()
       });
     }
     throw error;
   }
   ```

### Prevention Best Practices

1. **Validate Early in Development**:
   ```bash
   # Add to pre-commit hook
   bun run packages/scope-gate/bin/scope-validate.ts --path ./packages
   
   # Add to CI pipeline
   - name: Validate Scope Boundaries
     run: bun run packages/scope-gate/bin/scope-validate.ts
   ```

2. **Use Feature Flags Judiciously**:
   ```typescript
   // Enable only for specific environments
   if (process.env.NODE_ENV === 'development') {
     flagManager.enable('enable_experimental', 'Development testing');
   }
   
   // Set expiration for temporary flags
   flagManager.enable('enable_q2_feature', 'Q2 testing', {
     expiresAt: new Date('2024-06-30')
   });
   ```

3. **Document Scope Decisions**:
   ```markdown
   ## Scope Decisions
   
   ### Capability: knowledge-graph
   - **Scope Tag**: P1
   - **Rationale**: Advanced feature for post-V6.0
   - **Feature Flag**: enable_knowledge-graph
   - **Dependencies**: daemon-core, configuration
   - **Testing Notes**: Enable in development for integration testing
   ```

### Monitoring Best Practices

1. **Track Error Rates**:
   ```typescript
   // Monitor scope violation rate
   const violationRate = scopeViolations / totalRequests;
   if (violationRate > 0.01) { // 1% threshold
     alert('High scope violation rate detected');
   }
   ```

2. **Alert on Configuration Changes**:
   ```bash
   # Monitor .config.kiro changes
   git log --oneline -- .kiro/specs/*/.config.kiro
   
   # Alert on scope tag changes
   git diff HEAD~1 -- .kiro/specs/*/.config.kiro | grep scopeTag
   ```

3. **Regular Audits**:
   ```bash
   # Weekly audit script
   # 1. Check all feature flags
   bun run packages/scope-gate/bin/feature-flag.ts stats
   
   # 2. Validate all scope tags
   bun run packages/scope-gate/bin/scope-validate.ts --output json > audit-report.json
   
   # 3. Review violation history
   bun run packages/scope-gate/bin/feature-flag.ts history --output json > flag-history.json
   ```

## Support

If you encounter issues not covered in this guide:

1. **Check Logs**: Review application and audit logs for details
2. **Validate Configuration**: Run `scope-validate` to identify issues
3. **Review REQ-25**: Ensure capabilities are properly defined in parent spec
4. **Contact Maintainers**: File an issue in the SpecForge repository

**Remember**: Scope Gate errors are protective measures to ensure release boundary compliance. They indicate when code is attempting to use capabilities outside their approved scope, helping maintain architectural integrity across releases.
