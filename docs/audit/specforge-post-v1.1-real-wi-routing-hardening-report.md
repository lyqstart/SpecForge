## 1. 根因分析

### 症状

真实 OpenCode session 中输入"把保存按钮颜色调深一点"，未生成 v1.1 WI 结构（WI-xxxx/work_item.json），而是看到旧结构 `todo-list-web/spec.json`。

### 根因：Legacy 项目污染（非 runtime gap）

试运行项目 `D:\code\temp\wzq` 已有一个**预存的 legacy Work Item**：

```json
// .specforge/runtime/state.json
{
  "workItems": [{
    "work_item_id": "todo-list-web",
    "workflow_type": "feature_spec",
    "current_state": "implementation_ready"
  }]
}
```

```json
// .specforge/work-items/todo-list-web/spec.json
{
  "work_item_id": "todo-list-web",
  "workflow_type": "feature_spec",
  "current_state": "implementation_ready",
  "title": "Todo List Web 应用"
}
```

当 sf-orchestrator 启动时：
1. 检测到 `.specforge/project/spec_manifest.json` 存在 → 项目已初始化
2. 调用 `sf_state_read(work_item_id="all")` → 发现 `todo-list-web` 在 `implementation_ready`
3. 进入"会话恢复"流程 → 继续旧 WI，而非创建新的 v1.1 WI

**这不是 v1.1 主链路 bug**。sf-orchestrator 按设计执行了"已有进行中 WI → 尝试恢复"的逻辑。

### 为什么没有 `sf_v11_work_item_create` 工具

`sf_v11_work_item_create` 是 daemon-core 内部 handler，不是暴露给 OpenCode 的 tool。v1.1 设计中，WI 创建由 **orchestrator agent 直接写文件**（通过 `sf_artifact_write` 或内联写入），而非调用专用 tool。这是 by-design。

## 2. 正确的 Trial 流程

### 前提条件

真实 WI trial 必须在**干净项目目录**中执行：

```powershell
# 创建干净 trial 项目
mkdir D:\code\temp\sf-real-wi-trial-clean
cd D:\code\temp\sf-real-wi-trial-clean
git init
# 无需手动初始化 .specforge — daemon plugin 的 register 会触发 ensureProjectInit
```

### 或者清除旧 WI 状态

```powershell
# 在已有项目中清除旧 WI
Remove-Item "D:\code\temp\wzq\.specforge\work-items" -Recurse -Force
Remove-Item "D:\code\temp\wzq\.specforge\runtime\state.json" -Force
# 保留 manifest.json 和 config/
```

## 3. Session Binding Warnings 分析

```
[SessionRegistry] No session binding found for OpenCode event subtype: message.part.delta, projectPath: undefined
```

这些 warnings 来自 message/streaming 事件，它们不携带 `projectPath`。SessionRegistry 通过 sessionId→projectPath 映射来查找项目。如果 session 在 `register` 调用后已绑定，后续 message 事件应能通过 sessionId 找到绑定。

这些 warnings **不阻断 WI 创建**。它们只影响 per-event 的项目级路由（telemetry/logging），不影响 orchestrator 的工具调用（工具调用通过 `context.directory` 传递项目路径）。

## 4. 结论

| 需要修改代码 | ❌ 不需要 |
