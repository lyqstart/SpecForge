# 项目工程规则（通用基线）

<!-- 这是什么 -->
这份文件定义项目的工程纪律——所有 Agent 在生成代码、做审查、跑测试时
必须遵守的规则。它与技术栈无关，适用于所有项目。

技术栈相关的规则在 `languages/`、`frameworks/`、`databases/`、`infra/` 子目录中，
由 orchestrator 在 intake 阶段根据你的技术栈选择拼装。

<!-- 谁在什么时候用 -->
- **sf-executor**：写代码时遵守这些规则
- **sf-reviewer**：审查时机器 lint 这些规则（发现违反 = blocking）
- **sf-verifier**：测试时按这些规则验证

<!-- 怎么用 -->
1. 这份文件是所有项目的基线，通常不需要修改
2. 如果你的项目有特殊情况需要豁免某条规则，在规则后面加注释说明原因
3. 技术栈相关规则由 orchestrator 自动拼装，不要手动复制

---

## R1 配置管理（最重要）

**规则**：代码中不得硬编码以下任何值，必须通过配置文件或环境变量注入：

- IP 地址 / 主机名 / 域名
- 端口号
- 文件路径（绝对路径）
- 数据库连接串
- API URL / Webhook URL
- 超时秒数（业务相关的）
- 密码 / Token / API Key / 证书
- 功能开关（Feature Flag）

**为什么**：硬编码的值在不同环境（开发/测试/生产）之间不可复用，
且密钥一旦进入代码就会进入 git 历史，永远无法彻底删除。

**怎么做**：
- 配置文件：`config/app.yaml`（或项目约定的配置文件）
- 环境变量：`os.environ.get('KEY')` / `process.env.KEY`
- 密钥：使用 prod-environment.md 中 `credential_management` 指定的方式

**sf-reviewer 检查**：grep 代码中是否出现 IP 格式（`\d+\.\d+\.\d+\.\d+`）、
硬编码端口（`:8080`）、硬编码路径（`/opt/` / `C:\`）→ 发现 = blocking

---

## R2 依赖管理

**规则**：
1. 新增第三方依赖必须更新依赖声明文件（`package.json` / `requirements.txt` /
   `pom.xml` / `go.mod` / `Cargo.toml` / `*.csproj` 等）
2. 引入新依赖前必须在 task 描述中说明用途
3. 不得引入 license 不兼容的依赖（商业项目避免 GPL/AGPL）
4. 锁文件（`package-lock.json` / `poetry.lock` / `go.sum` 等）必须提交到 git

**为什么**：未声明的依赖会导致其他人无法复现构建；
license 不兼容可能带来法律风险。

**sf-reviewer 检查**：diff 中有新 import 但依赖文件未更新 → blocking

---

## R3 版本兼容

**规则**：新代码必须在 `prod-environment.md` 中 `runtimes.*_min` 指定的
最低版本通过编译/lint，不得使用超出该版本的语法或库特性。

**为什么**：开发机版本通常高于生产版本，用了新语法在生产会报错。

**sf-verifier 检查**：L9 兼容性测试在生产最低版本跑一遍。

---

## R4 代码修改纪律

**规则**：
1. 一个 task 只改一个目的；diff 中每行变更必须直接服务于 task 描述
2. 不得顺手改格式（空行/引号/缩进）——格式改动单独立 task
3. 不得顺手重命名变量/函数——重命名单独立 task
4. 不得在修复 bug 时顺手重构——重构单独立 task

**为什么**：混入无关改动会让 code review 变难，也会让 git blame 失去意义。

---

## R5 风格一致性

**规则**：新文件的风格必须匹配相邻文件（缩进/命名/字符串引号/换行符/文件末尾换行）。

**怎么做**：写代码前先读 2-3 个相邻文件，观察风格后再写。

**sf-reviewer 检查**：新文件与相邻文件风格不一致 → warning

---

## R6 安全基线

**规则**：
1. 不得将未经 sanitize 的用户输入直接拼接到 SQL / Shell 命令 / HTML
2. 不得在日志中打印密码 / Token / API Key / 个人身份信息（PII）
3. 不得在代码注释中写密码或密钥（即使是"示例"）
4. 对外暴露的接口必须有认证（除非明确设计为公开接口）

**为什么**：SQL 注入、XSS、日志泄露是最常见的安全漏洞，
且一旦进入 git 历史就很难彻底清除。

**sf-reviewer 检查**：发现 SQL 拼接 / 日志打印敏感字段 → blocking

---

## R7 错误处理

**规则**：
1. 所有外部 IO（HTTP 请求/数据库/文件读写/子进程）必须显式处理失败
2. 不得用空 catch 块吞掉异常（`catch (e) {}`）
3. 错误必须有足够的上下文信息（"什么操作失败了"+"失败原因"）
4. 用户可见的错误消息不得暴露内部实现细节（堆栈/SQL/文件路径）

**为什么**：吞掉异常会让问题在生产环境悄悄发生，难以排查。

---

## R8 日志规范

**规则**：
1. 使用 `prod-environment.md` 中 `log_framework` 指定的日志框架
2. 不得用 `print` / `console.log` 在生产代码中输出（测试代码除外）
3. 日志级别严格使用：
   - DEBUG：调试信息（生产不输出）
   - INFO：正常业务事件（服务启动/请求完成）
   - WARN：可恢复的异常（重试/降级）
   - ERROR：需要人工介入的错误
4. 日志格式使用 `prod-environment.md` 中 `log_format` 指定的格式

**为什么**：统一日志框架和格式便于集中收集和分析；
乱用 print 会让日志系统失效。
