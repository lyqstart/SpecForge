# Intake 阶段提问脚本

> 本文件由 sf-orchestrator 在 intake 阶段加载。
> 它定义了 orchestrator 在 intake 阶段必须完成的两件事：
> 1. 收集用户需求（所有 WI 都做）
> 2. 首次 WI 时：收集技术栈决策，生成三份配置文件

---

# 阶段 A：开发环境初始化（仅首次，或 dev-environment.md 不存在时）

## A1 检测与版本验证

```
检测 .specforge/ 目录：
  不存在 → 创建目录
  存在 → 读取 manifest.json 中的 schema_version
         < v6.0 → 停止，提示用户：
                  "当前项目使用的是旧版 SpecForge 配置（v{version}），
                   与 V6.0 不兼容。请先运行迁移工具或手动清理 .specforge/ 目录后重试。"
         ≥ v6.0 → 继续
```

## A2 开发环境扫描与确认

```
检测 .specforge/dev-environment.md：

情况 1：不存在
  1. 运行 scan-host-profile.ts 扫描本机
  2. 把扫描结果填入 dev-environment 模板
  3. 展示给用户：
     "✅ 已扫描到以下开发环境信息，请确认：
      - 操作系统：{os.version}
      - 主要语言：{runtimes 列表}
      - Shell：{shell.preferred}
      - 时区：{locale.tz_name}（{locale.system_lang}）
      - 网络：{network.has_internet ? '有外网' : '无外网'}
      [y] 确认  [n] 手动编辑  [s] 跳过（稍后手动填）"
  4. 用户确认后写入 .specforge/dev-environment.md

情况 2：已存在
  1. 运行 scan-host-profile.ts 重新扫描
  2. 与现有文件比对
  3. 如有差异，逐项展示：
     "⚠️ 检测到开发环境变化：
      - node 版本：18.0.0 → 20.11.0
      - 新增工具：bun 1.1.0
      是否更新配置？[y] 全部更新  [n] 保持原样  [s] 逐项选择"
  4. 按用户选择更新文件

情况 3：存在且无差异
  → 跳过，直接进入 B 阶段
```

---

# 阶段 B：需求与技术栈收集（每个 WI 的 intake）

## B0 检查已有配置

```
检测 .specforge/prod-environment.md 和 .specforge/project-rules.md：

两个文件都存在：
  展示摘要：
  "📋 项目已有以下配置：
   - 生产环境：{deployment_target} / {os.platform}
   - 主语言：{runtimes 列表}
   - 工程规则：{project-rules.md 的技术栈摘要}
   是否沿用？[y] 沿用  [n] 重新配置"
  用户选 y → 跳到 B1（只收集需求）
  用户选 n → 进入 B2（重新收集技术栈）

只有部分文件存在：
  → 提示缺少的文件，进入对应的收集步骤

两个文件都不存在（首次 WI）：
  → 进入 B1
```

## B1 收集用户需求（所有 WI 必做）

orchestrator 必须覆盖以下 10 个维度，**一次问完，不分批**：

```
📋 请告诉我关于这个项目/功能的信息：

1. 业务功能
   "你想要做什么？请描述核心功能和业务目标。"

2. 用户群体
   "谁会使用这个系统？"
   选项：
   [a] 个人使用（只有我自己用）
   [b] 小团队（5-20 人）
   [c] 企业内部（公司员工）
   [d] 对外发布的产品（面向公众）
   [e] 开源项目

3. 使用规模
   "预计有多少用户？数据量大概多少？"
   例：100 人 / 1 万人 / 100 万人；1GB 数据 / 1TB 数据

4. 并发与性能要求
   "对响应速度有要求吗？同时使用的人多吗？"
   例：无特殊要求 / P95 < 500ms / 支持 1000 并发

5. 可用性要求
   "如果系统挂了，影响有多大？"
   选项：
   [a] 个人工具，偶尔挂一下没关系
   [b] 内部工具，工作时间可用就行（99%）
   [c] 商业服务，需要高可用（99.9%+）
   [d] 关键业务，不能停（99.99%+）

6. 安全合规要求
   "有没有特殊的安全或合规要求？"
   例：无 / 需要保护个人隐私（GDPR/个保法）/ 金融合规 / 医疗合规

7. 未来打算
   "这个项目的长期计划是什么？"
   选项：
   [a] 一次性工具，用完就扔
   [b] 长期维护，但功能相对稳定
   [c] 持续演进，功能会不断增加
   [d] 计划开源
   [e] 计划商业化

8. 团队情况
   "谁来开发和维护？"
   选项：
   [a] 只有我一个人
   [b] 2-5 人小团队
   [c] 10 人以上团队

9. 预算约束
   "对基础设施成本有限制吗？"
   例：免费方案 / 月 $50 以内 / 月 $500 以内 / 无限制

10. 其他约束
    "有没有其他重要约束？"
    例：必须用某种语言 / 必须部署在某个平台 / 必须兼容某个系统
```

## B2 推荐技术栈（首次 WI 或用户选择重新配置时）

基于 B1 的回答，orchestrator 给出 2-3 套候选方案：

```
推荐格式：

🔧 方案 A：[方案名称]
  - 主语言/框架：[具体技术]
  - 数据库：[具体技术]
  - 缓存：[具体技术 / 无]
  - 消息队列：[具体技术 / 无]
  - 部署方式：[具体方式]
  - CI/CD：[具体平台]
  ✅ 适合：[适用场景描述]
  ⚠️ 注意：[潜在风险或限制]

🔧 方案 B：...

🔧 方案 C：...

请选择方案（A/B/C），或告诉我你的偏好：
```

**推荐时必须考虑的 10 个维度**：
1. 需求规模匹配（小→ SQLite + 单服务；大→ PostgreSQL + 微服务）
2. 团队熟悉度（已知技术栈 vs 学习成本）
3. 生态成熟度（首选有大量参考实现的栈）
4. 性能/资源契合度
5. 可维护性/招聘难度（团队场景）
6. 部署复杂度（用户偏好简单 vs 接受 K8s）
7. 许可证（商业用途必须避开 GPL/AGPL）
8. 长期演进风险（避免冷门栈）
9. 与已有系统的集成
10. 用户明示偏好（如"不要 Java"）

## B3 技术栈决策细节（用户选定方案后）

按选定技术栈，逐字段问决策细节。**只问相关字段，不相关的跳过**：

```
对于每个相关字段：
  "X 的版本/配置是什么？"
  推荐值：[默认推荐值]（[推荐理由]）
  [Enter] 使用推荐值  [输入] 自定义值

例：
  "Python 最低版本？"
  推荐值：3.10（支持 match/case，2026 年前有维护）
  [Enter] 使用 3.10  [输入] 自定义

  "数据库端口？"
  推荐值：5432（PostgreSQL 默认）
  [Enter] 使用 5432  [输入] 自定义
```

**必须覆盖的字段类别**（按选定技术栈筛选）：
- 语言最低版本（生产）
- 数据库连接信息（host/port/name，不含密码）
- 缓存连接信息（如有）
- 部署目标详情（服务器 IP/域名/SSL）
- 日志框架选择
- 测试框架选择
- CI/CD 平台

## B4 生成三份配置文件

```
基于 B1-B3 的回答：

1. 从 ~/.specforge/templates/ 选择对应模板：
   - prod-environment 模板：按部署目标选
   - project-rules 模板：_BASE + 主语言 + 框架 + 数据库 + infra 拼装

2. 填入用户的决策值

3. 写入：
   - .specforge/prod-environment.md
   - .specforge/project-rules.md

4. 展示摘要让用户确认：
   "✅ 已生成以下配置：
    - 生产环境：{deployment_target} / Python {python_min}
    - 工程规则：Python + FastAPI + PostgreSQL + GitHub Actions
    是否确认？[y] 确认  [n] 修改"
```

## B5 技术栈最佳实践提醒

```
基于选定技术栈，主动告知用户：

"📌 基于你选择的技术栈，建议同时做以下事情：

1. .gitignore 建议添加：
   [列出对应技术栈的 .gitignore 条目]

2. 推荐目录结构：
   [列出对应技术栈的目录结构]

3. 推荐同时配置：
   [列出对应技术栈的配套工具]

是否需要我帮你创建这些文件？[y] 是  [n] 稍后自己处理"
```

## B6 进入 requirements 阶段

```
intake 完成后：
  - 三份配置文件已生成（或已确认沿用）
  - 需求信息已收集（B1 的 10 个维度）
  - 写入 .specforge/specs/{WI-ID}/intake.md

调度 sf-requirements：
  - 传入 intake.md（需求信息）
  - 不传入技术栈信息（sf-requirements 不需要，需求与技术栈无关）
```

---

# 注意事项

1. **需求先于技术栈**：必须先完成 B1（收集需求），再做 B2（推荐技术栈）。
   不得在不了解需求的情况下推荐技术栈。

2. **技术栈是项目级的**：一旦确定，整个项目所有 WI 都用同一套技术栈。
   不会随着 WI 变化而变化。

3. **sf-requirements 不看技术栈**：需求文档描述"做什么"，
   技术栈是"怎么做"，属于 sf-design 的决策范围。

4. **配置文件不含密码**：所有密码/密钥通过环境变量或密钥管理服务注入，
   不写入 .specforge/ 目录下的任何文件。
