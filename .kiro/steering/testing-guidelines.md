# 测试规范

> 从真实 bug 中提炼的测试原则。适用于任何项目。

---

## 一、Mock 纪律

### 1.1 不要 mock 掉被测对象

如果函数 A 调用函数 B，你在测试 A 时可以 mock B。但如果你在测试 B 本身，绝不能 mock B。

"难以 mock"不是跳过测试的理由——它是需要直接测试的信号。

### 1.2 Mock 边界必须与真实边界一致

Mock 的行为必须与真实实现的契约一致。如果真实函数在特定平台上返回不同格式的值（如 Windows 路径带前导 `/`），mock 也必须反映这一点，否则测试就是在验证一个不存在的世界。

### 1.3 每个 mock 都应该有一个对应的"无 mock"测试

对于关键路径（启动流程、安装流程、状态流转），至少有一个集成测试不使用任何 mock，在真实文件系统上运行。

---

## 二、失败路径覆盖

### 2.1 静默跳过必须被测试

代码中每个 `continue`、空 `catch {}`、`return` 跳过的地方，测试中必须有用例验证：
- 跳过条件是否正确触发
- 跳过后的最终状态是否符合预期
- 累积跳过是否导致整体操作无效（如部署 0 个文件仍报告成功）

### 2.2 错误传播 vs 错误吞没必须明确

对于每个 catch 块，测试必须验证：
- 该错误是被重新抛出（调用方能感知）还是被吞掉（调用方无感知）
- 如果吞掉，是否有日志/指标记录
- 日志写入本身失败时，是否会导致二次异常

### 2.3 "成功"的定义必须严格

操作报告"成功"时，测试必须断言最终状态满足所有后置条件，而不仅仅是"没有抛异常"。

---

## 三、平台与环境

### 3.1 路径处理必须有平台断言

涉及文件路径的函数，测试必须断言返回值在当前平台上有效：
- 路径指向的位置 `existsSync()` 为 true（或明确预期不存在）
- 不包含畸形模式（如 `D:\d:\...`、双斜杠 `//`）
- Windows 上使用 `fileURLToPath()` 而非手动解析 `URL.pathname`

### 3.2 环境变量和配置的边界值

测试必须覆盖：
- 环境变量未设置时的默认行为
- 环境变量为空字符串时的行为
- 配置文件缺失、为空、格式错误时的行为

---

## 四、状态一致性

### 4.1 操作完成后必须验证最终状态

对于改变文件系统、数据库或状态文件的操作，测试必须在操作完成后断言最终状态的完整性，而不仅仅验证中间步骤。

```
// ❌ 只验证了"文件被写入"
expect(writeFileCalled).toBe(true)

// ✅ 验证最终状态
expect(existsSync(targetPath)).toBe(true)
expect(readFileSync(targetPath, "utf-8")).toEqual(expectedContent)
```

### 4.2 幂等性必须被测试

以下场景必须有覆盖：
- 同一操作连续执行两次 → 结果一致
- 操作中断后重新执行 → 能恢复到正确状态
- 升级操作后 → 不残留旧版本文件

### 4.3 并发和竞态

涉及锁、原子写入、多进程的代码：
- 锁获取失败时的行为
- 原子写入中途崩溃后的恢复
- 两个进程同时操作同一资源

---

## 五、验证策略分层

### 5.1 单元测试：验证逻辑正确性

- 纯函数的输入输出映射
- 边界值、空值、非法输入
- 不涉及文件系统或网络

### 5.2 集成测试：验证组件协作

- 真实文件系统操作（使用 tmpdir）
- 多个模块组合调用的端到端路径
- 配置文件解析 → 行为变化的完整链路

### 5.3 Property-Based Testing：验证不变量

适用于有明确正确性属性的场景：
- 解析器：`parse(serialize(x)) === x`（往返一致性）
- 状态机：任何合法输入序列不会导致非法状态
- 日志轮转：轮转后文件编号严格递增、总数不超过上限
- 错误处理：任何异常输入不会导致未捕获异常逃逸

### 5.4 冒烟测试：验证部署产物可用

安装/升级/构建完成后，执行最小可用性检查：
- 所有注册的文件都存在
- 关键入口函数可以被 import 且不抛异常
- 配置文件可以被解析

---

## 六、测试编写检查清单

编写测试前，回答以下问题：

1. **被测函数的核心逻辑是否被直接测试？** 还是被 mock 绕过了？
2. **失败路径是否有覆盖？** 每个 catch/continue/early-return 是否有对应用例？
3. **最终状态是否被断言？** 还是只验证了"没报错"？
4. **跨平台行为是否被考虑？** 路径、换行符、文件权限在 Windows/Linux 上是否一致？
5. **操作的幂等性是否被验证？** 重复执行是否安全？
6. **如果这个测试在 mock 环境中通过但真实环境中失败，我能发现吗？** 如果不能，需要补充集成测试。
